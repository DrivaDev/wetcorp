## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (Tailwind `gap-1`, `p-1`) | Icon gaps, badge padding |
| sm | 8px (Tailwind `gap-2`, `p-2`) | Inline element spacing, button icon gaps |
| md | 16px (Tailwind `gap-4`, `p-4`) | Default element spacing, card inner padding |
| lg | 24px (Tailwind `gap-6`, `p-6`) | Card padding, section spacing |
| xl | 32px (Tailwind `gap-8`, `p-8`) | Layout column gaps |
| 2xl | 48px (Tailwind `py-12`) | Major section breaks, empty state vertical padding |
| 3xl | 64px (Tailwind `py-16`) | Page-level empty state vertical padding |

Exceptions:
- Touch targets: `min-h-[44px] min-w-[44px]` on all icon buttons (WCAG 2.5.5) — pre-established pattern from Phase 2/3
- Skeleton bar heights: 16px (`h-4`), 20px (`h-5`), 32px (`h-8`) — match real content heights
- Badge vertical padding: `py-0.5` (2px) — pre-established in OCTable badge system

Source: established patterns from Phase 2 (OCTable, StatCard) and Phase 3 (GastosCard, WizardPage).

Source: CLAUDE.md (Fira Sans weight scale 700/500/400/300). Confirmed against ResumenStep1.tsx, GastosCard.tsx, ValueCards.tsx patterns.

## Page Layouts

### 1. OC Detail Page — All Roles

**Routes:**
- `/importador/oc/[id]` — importador role
- `/proveedor/oc/[id]` — proveedor role
- `/despachante/oc/[id]` — despachante role

**Page structure** (Server Component `page.tsx`):

```
<page bg-fondo min-h-screen>
  <OCDetailHeader />          ← breadcrumb + OC number + EstadoBadge + action buttons
  <main px-4 sm:px-8 py-6 max-w-5xl mx-auto flex flex-col gap-6>
    <ResumenStep1 />          ← read-only: Datos OC + Productos (reuse from wizard)
    <GastosCard />            ← Despacho section, read-only (onChange = noop)
    <GastosCard />            ← Despachante section, read-only
    <GastosCard />            ← Adicionales section, read-only
    <OtrosGastosSection />    ← read-only (no add/remove)
    <ValueCards />            ← FOB / Gastos / Landed / Impuestos
    <DocumentSlots />         ← read-only (no upload buttons for proveedor/despachante)
  </main>
</page>
```

**Read-only mode for GastosCard:** pass `onChange={() => {}}` and render values as display text. Inputs MUST be replaced with `<p>` elements styled with `readOnlyClass` = `text-base font-normal text-texto bg-fondo px-3 py-2 rounded-lg border border-acento/50` (same class as ResumenStep1.tsx).

**Read-only mode for DocumentSlots:** in detail view, hide Upload/Cambiar buttons. Show document name + `"Sin archivo adjunto"` or filename as static text. Proveedor/despachante see download link if URL exists (Phase 6 will wire real URLs; Phase 4 shows placeholder state).

### 2. OC Edit Page — All Roles

**Routes:**
- `/importador/oc/[id]/editar`
- `/proveedor/oc/[id]/editar`
- `/despachante/oc/[id]/editar`

**Page structure:** Mount `<WizardPage />` as-is. The page's `useEffect` pre-loads `sessionStorage['oc-step1-draft']` from `OCDetalle` mock data before rendering the wizard. Visually identical to `/importador/oc/nueva`.

**Pre-load pattern (client component wrapper):**
```
<EditWizardLoader ocId={id}>  ← 'use client', reads OCDetalle from mock, writes sessionStorage, renders WizardPage
  <WizardPage step={1} />
</EditWizardLoader>
```

## Read-Only Mode Visual Treatment

When reusing wizard components in the detail view, inputs become display elements. Rules:

| Original element | Read-only replacement | Tailwind classes |
|------------------|-----------------------|-----------------|
| `<input type="text">` | `<p>` | `text-base font-normal text-texto bg-fondo px-3 py-2 rounded-lg border border-acento/50` |
| `<input type="number">` | `<p>` | same as above |
| `<select>` | `<p>` | same as above |
| `<textarea>` | `<p>` | `text-base font-normal text-texto bg-fondo px-3 py-2 rounded-lg border border-acento/50` |
| Upload button | hidden | n/a |
| Add row button | hidden | n/a |
| Remove row button | hidden | n/a |

**Label style (above each read-only field):** `text-sm font-normal text-titulares mb-1`

Source: `readOnlyClass` constant already defined in `ResumenStep1.tsx` — use this exact string across all read-only components.

**GastosCard in read-only mode:** The existing `onChange` prop is required. Pass `() => {}` (noop). However, the input elements themselves must be replaced with `<p>` elements. This means GastosCard needs a `readOnly?: boolean` prop that switches between input and display rendering. Do NOT mutate existing behaviour — add the prop as opt-in.

## Interaction Contracts

### OCDetailHeader — Action Buttons

| Button | Role visibility | Interaction | Destination |
|--------|----------------|-------------|-------------|
| Editar | importador + proveedor + despachante | Link click | `/{rol}/oc/{id}/editar` |
| Eliminar | importador ONLY | Button click → opens DeleteModal | Modal → mock delete → redirect to dashboard |

### DeleteModal (existing component)

Existing `src/components/dashboard/DeleteModal.tsx` is reused from the detail header. No changes needed to the modal itself.

### Edit Mode Pre-load

1. `EditWizardLoader` ('use client') receives `ocId`
2. On mount (`useEffect`), reads `OCDetalle` from `MOCK_OCS_DETALLE` by `ocId`
3. Maps `OCDetalle` fields to `Step1Data` shape and writes to `sessionStorage['oc-step1-draft']`
4. Renders `<WizardPage step={step} />` where `step` comes from URL `searchParams`
5. If OC not found: calls `router.replace('/{rol}/dashboard')` — does not show 404

### Loading State

All data fetching is synchronous mock in Phase 4 (no async). However, components that previously accepted `isLoading` prop (OCTable) already handle skeletons. For the new OC detail page, the skeleton is rendered as a static loading component only if the page architecture introduces a Suspense boundary or a simulated delay for demo purposes.

**Default for Phase 4:** No artificial delay. Skeleton components are built but only activated if the planner determines a `loading.tsx` file is needed per Next.js App Router convention.
